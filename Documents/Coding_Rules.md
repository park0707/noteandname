# Coding Rules — 노벨플로우 코딩 규칙 문서

> 이 문서는 개발 과정에서 실제로 겪은 문제와 해결책, 지켜야 할 코딩 관행을 기록합니다.  
> 에이전트는 **작업 시작 전 반드시 이 문서를 읽고** 동일한 실수를 반복하지 않습니다.  
> 새로운 문제나 해결책이 생기면 이 문서에 지속적으로 추가합니다.

---

## 🚫 하지 말아야 할 것 (Anti-Patterns)

### [일반]

- **지시 범위 외 코드 수정 금지**  
  요청과 직접 관련 없는 코드(개선 여지가 보이더라도)를 임의로 수정하면 안 됩니다.  
  → 수정이 필요하다고 판단되면 사용자에게 이유를 먼저 설명하고 허가를 받습니다.

- **사용자가 작성한 코드 허가 없이 변경 금지**  
  사용자가 직접 수정한 코드는 명시적 승인 없이 변경 불가합니다.  
  → 단, 해당 코드가 빌드를 파괴하는 치명적 컴파일 에러를 유발하는 경우에만 최소 처리 후 즉시 보고합니다.

- **대규모 블록 치환 금지**  
  수백 줄에 달하는 코드를 한 번에 통째로 치환하면 기존에 정상 작동하던 로직이 사라지는 사고가 납니다.  
  → 반드시 줄 단위로 정밀하게 비교·대체하는 Surgical Edit 방식을 사용합니다.

- **추측으로 수정하지 않기**  
  에러 메시지를 보지 않고 짐작으로 코드를 고치면 새로운 버그가 생깁니다.  
  → 반드시 터미널 출력과 실제 파일 내용을 확인한 후 수정합니다.

- **영향 범위 조사 없는 공통 모듈/상태/유틸리티 수정 금지**  
  공통 훅(`useEditorFormat`, `useEpisodes` 등), 공통 유틸리티, 공유 Props/Type, 전역 상태(Zustand)를 고칠 때 호출부 전수 조사를 건너뛰면 타 기능이 연쇄 파괴됩니다.

- **기존 Props 및 API 인터페이스 파괴적 변경(Breaking Change) 금지**  
  기존 매개변수를 제거하거나 필수 타입으로 수정하여 기존 호출처에서 컴파일/런타임 에러를 유발하면 안 됩니다.  
  → 선택적 프로퍼티(`?`) 지정 및 기본값(Default Value) 설정이 필수입니다.

- **특정 타입 전용 로직을 공통 흐름에 분기 조건 없이 섞기 금지**  
  `handleCanvasMouseMove`나 `focusOnElement` 등 공통 이벤트 핸들러에 특정 요소(예: `image` 또는 `brush`) 전용 로직을 작성할 때 타입 분기 guard를 두지 않으면 다른 요소(다각형, 핀, 교역로 등) 작동이 마비됩니다.

- **새 기능 구현/버그 수정과 대규모 구조 리팩토링 혼합 금지**  
  작업 목적과 직접 관련 없는 파일 리팩토링이나 구조 변경을 한 번에 진행하면 버그의 원인 추적이 불가능해집니다.

### [TypeScript / React]

- **`transition-all duration-200` 을 드래그 요소에 사용 금지**  
  드래그 중 좌표가 매 프레임 갱신될 때 CSS 트랜지션이 200ms 보간을 재설정하면서  
  요소가 마우스 커서를 못 따라오는 고무줄 현상(Rubber-banding)이 발생합니다.  
  → 드래그하는 요소에는 `transition: 'none'`을 명시적으로 적용합니다.

- **SVG `<image>` 요소에 base64 Data URL을 직접 href로 사용 금지**  
  드래그 시 매 프레임마다 대용량 base64 문자열을 재파싱·재래스터화하여 CPU 사용량이 급증하고 프레임 드롭이 발생합니다.  
  → `URL.createObjectURL(Blob)`으로 변환하여 브라우저 Decoded Image Cache를 활용합니다.

- **SVG 요소의 `x`, `y` 속성을 드래그 좌표로 직접 갱신 금지**  
  SVG `x`, `y` 속성을 변경하면 Layout(Reflow)이 유발되어 CPU 래스터화 비용이 발생합니다.  
  → `<g style="transform: translate(bx px, by px)">` 래퍼를 사용하여 GPU Composite 레이어로 이동합니다.

- **Map 제네릭 타입 TypeScript 선언 시 `new Map()` 주의**  
  ```ts
  // ❌ 컴파일 에러: Expected 1 arguments, but got 0
  const ref = useRef<Map<string, string>>(new Map());

  // ✅ 올바른 방법: 일반 객체 사용
  const ref = useRef<{ [key: string]: string }>({});
  ```

- **`const [, setter] = useState()` 패턴 — 미사용 state 값 처리**  
  state 값을 선언하되 읽지 않을 경우 TypeScript가 `TS6133: declared but never read` 에러를 냅니다.  
  ```ts
  // ❌ 에러 발생
  const [hoveredElementId, setHoveredElementId] = useState<string | null>(null);

  // ✅ 첫 번째 요소를 콤마로 생략
  const [, setHoveredElementId] = useState<string | null>(null);
  ```

### [SVG / 렌더링 성능]

- **이미지 요소의 스냅(Snap) 좌표 적용 금지**  
  이미지(type: 'image') 요소를 리사이즈할 때 격자 스냅이 적용된 snapped 좌표를 사용하면  
  크기가 계단식으로 툭툭 끊기는 현상이 발생합니다.  
  → 이미지 리사이즈 시에는 스냅이 없는 순수 물리 마우스 좌표(`currentCoords`)를 사용합니다.

- **이미지 드래그 시 점간(Node) 자석 스냅 적용 금지**  
  이미지 요소를 드래그할 때 `pointSnapEnabled`가 켜져 있으면 주변 핀·꼭짓점에 달라붙어 뚝뚝 끊깁니다.  
  → 드래그 대상이 이미지 전용일 때는 점간 스냅 연산을 건너뜁니다.

## ✅ 해야 할 것 (Best Practices)

### [작업 시작 전]

- 작업 전 반드시 `Documents/Coding_Rules.md`(이 파일)를 읽습니다.
- 작업 전 연관된 `Documents/` 내 설계서·기획서를 확인합니다.
- 수정 범위(영향받는 파일, 타입 시스템, 의존 관계)를 먼저 파악합니다.

### [작업 중]

- **컴파일 무결성 우선**: 파일 수정 후 반드시 `npx tsc -b` → `npm run lint` 순서로 검증합니다.
- **Self-Healing 루프**: 에러 발생 시 터미널 로그 → 파일 확인 → 수술식 수정 → 재검증 사이클을 반복합니다.
- **비파괴적 인터페이스 유지**: Props 변경 시 기존 호출부가 깨지지 않도록 선택적 프로퍼티(`?`) 또는 기본값을 우선 사용합니다.
- **오픈소스 라이선스 준수 및 상업용 검증 규정**:
  1. **상업적 이용 불가능 라이브러리 사용 절대 금지**: GPL, AGPL, CC-BY-NC 등 상업적 이용을 제한하거나 소스코드 전면 공개를 강제하는 라이선스는 절대 도입하지 않으며, 반드시 상업적 이용이 가능한 무료 오픈소스(MIT, ISC, Apache 2.0, BSD, SIL OFL 1.1)만 사용합니다.
  2. **출처 표기(Attribution) 조건 자동 충족**: 저작권 명시(Notice) 의무가 있는 오픈소스 라이브러리 사용 시, 앱 내 정보/설정 모달 또는 서비스 라이선스 문서에 저작권자 표기(Attribution)를 자동으로 명시하여 법적 조건을 완벽히 충족합니다.

### [🛡️ 코드 수정 시 부작용(Side Effect) 및 타 기능 파급 예방 5단계 수칙]

1. **1단계: 영향 범위 전수 조사 (Search Before Modifying)**
   - 공통 훅, 유틸리티 함수, 공유 Props, TypeScript 타입, 전역 상태(Zustand)를 수정할 때는 반드시 워크스페이스 전역 검색(`grep_search` 또는 IDE 참조 찾기)을 통해 **해당 대상을 사용하는 모든 호출처**를 100% 식별합니다.
   - 단 한 곳의 호출처라도 놓치면 해당 기능에서 런타임/타입 에러가 발생합니다.

2. **2단계: 인터페이스 비파괴적 확장 (Defensive Interface Design)**
   - 기존 Props, State, API 함수 매개변수를 함부로 삭제하거나 필수(required) 타입으로 변경하지 않습니다.
   - 매개변수를 추가할 때는 항상 선택적 프로퍼티(`?`) 또는 기본값(Default value)을 선언하여 **기존 호출처 코드의 호환성을 100% 보장**합니다.

3. **3단계: 가드 클로저 및 타입 분기 격리 (Guard Clauses & Feature Isolation)**
   - 이벤트 핸들러(`handleCanvasMouseMove`, `focusOnElement` 등)나 공통 연산 로직에 특정 요소/기능 전용 코드(예: `image` 전용 리사이즈, `brush` 전용 자석 스냅 제어)를 작성할 때는 **Guard Clause (`if (element.type !== 'target') return;`)** 또는 **명확한 타입 분기**를 적용합니다.
   - 다른 요소/기능의 실행 경로(Execution Path)를 절대 침범하거나 왜곡하지 않도록 로직을 완벽히 격리합니다.

4. **4단계: 점진적 수술적 수정 (Surgical Edit & Atomic Changes)**
   - 수정이 필요한 좁은 범위만 정밀 대체(Surgical Edit)하며, 지시받지 않은 주변 로직/구조 개편을 한 번에 진행하지 않습니다.
   - 한 번의 수정 작업에서 여러 개의 서로 다른 요구사항이나 리팩토링을 섞어 다루지 않습니다.

5. **5단계: 빌드 무결성 검증 및 교차 검증 (Cross-Feature Verification)**
   - **빌드 검증**: 코드 수정 직후 `npx tsc -b`를 실행하여 컴파일 무결성을 입증합니다.
   - **교차 시나리오 검증**: 수정한 기능뿐만 아니라, **동일한 핸들러/상태를 공유하는 다른 7대 표준 요소 및 타 기능**이 정상 작동하는지 반드시 연계 테스트(드래그, 클릭, 선택, 스냅, 잠금, 줌 등)를 수행합니다.

### [📋 코드 수정 시 회귀 방지 체크리스트 (Regression Prevention Checklist)]

수정 작업을 완료하기 전, 아래 6가지 항목을 100% 확인합니다:

| 번호 | 체크 항목 | 검증 여부 |
|:---:|:---|:---:|
| 1 | 워크스페이스 전체 조사를 통해 수정한 모듈/함수/타입의 **모든 호출처**를 확인했는가? | [ ] |
| 2 | Props/Interface 변경 시 기존 호출부가 깨지지 않도록 **선택적 프로퍼티(`?`)나 기본값**을 적용했는가? | [ ] |
| 3 | 공통 핸들러 내 특정 타입/기능 전용 로직에 **Guard Clause / 타입 분기**가 완벽히 적용되었는가? | [ ] |
| 4 | 사용자 수정 코드 및 요청 범위를 벗어난 임의 코드를 건드리지 않았는가? | [ ] |
| 5 | `npx tsc -b` 실행 결과 **타입 오류(0 에러)**를 확인하였는가? | [ ] |
| 6 | 수정한 기능과 동일한 공통 핸들러/상태를 공유하는 **연관 기능들이 정상 작동**하는지 교차 확인했는가? | [ ] |

### [SVG 요소 드래그·리사이즈]

- 드래그하는 요소는 `<g style={{ transform: 'translate(x px, y px)', willChange: isDragging ? 'transform' : 'auto' }}>` 패턴을 사용합니다.
- 이미지 요소는 드래그 중 `transition: 'none'`을 반드시 인라인 스타일로 지정합니다.
- 이미지 `href`에는 base64 대신 `URL.createObjectURL()` 캐시를 사용합니다.
- 리사이즈 핸들 앵커는 SVG 전체 좌표계(절대 좌표 `bx, by`)를 기준으로 렌더링합니다.  
  (`<g transform>` 방식 사용 시 자식 요소는 로컬 좌표 `0, 0` 기준)

### [아이콘 및 UI 용어 통일 규정]

- **윈도우/OS 텍스트 이모지(Emoji) UI 사용 금지**:
  - `🖼️`, `🗺️`, `📍`, `🎨`, `🔳`, `⚪`, `🛣️` 등 브라우저/OS 환경에 따라 디자인 일관성을 해치는 텍스트 이모지를 UI에 절대 사용하지 않습니다.
- **Lucide React SVG 아이콘 컴포넌트 단일화**:
  - 상업적 이용이 가능한 무료 오픈소스 라이브러리 `lucide-react` 아이콘을 단일 표준으로 사용합니다.
- **지도 요소 표준 용어 및 아이콘 표**:
  - `image`: **배경 이미지** (`<Image />`)
  - `polygon`: **다각형 영역** (`<Map />`)
  - `route`: **이동 교역로** (`<Route />`)
  - `brush`: **붓 그리기 영역** (`<Paintbrush />`)
  - `pin`: **거점 핀 마커** (`<MapPin />`)
  - `border_rect`: **구역 사각형 테두리** (`<Square />`)
  - `border_circle`: **구역 원형 테두리** (`<Circle />`)

### [사이드바 요소 연동]

- 새로운 `MapElement` 타입을 추가할 때 반드시 아래 항목을 함께 업데이트합니다:
  1. `MapElement` 타입 유니온 추가
  2. `FlatTreeNode` 타입 유니온 추가
  3. `selectedSidebarTypes` 기본값 배열에 추가
  4. 사이드바 아이콘 맵에 Lucide SVG 아이콘 추가
  5. 사이드바 필터 팝업 체크박스 추가
  6. `focusOnElement` 함수에 타입 분기 추가 (클릭 시 해당 위치로 이동)
  7. 드래그 이동 및 리사이즈 이벤트 핸들러에 타입 포함 여부 확인
