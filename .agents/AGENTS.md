# Agent Coding Rules & Guidelines (노벨플로우 워크스페이스 코딩 규칙)

에이전트는 사용자의 지시를 수행할 때 아래의 **4단계 프로세스 및 세부 코딩 규칙**을 철저히 준수해야 합니다.

---

## 1. 개발 프로세스 4단계 규정

### 1단계: 관련 문서 탐색 및 계획 수립
*   사용자로부터 지시 사항을 전달받으면, 작업 착수 전에 반드시 연관된 설계 사양서, 참고 문서(`Documents/` 폴더 내 마스터 기획 등), 기존 코드 흐름을 분석합니다.
*   **Documents 필수 열람 규정**: 작업 시작 전 반드시 다음 두 가지를 먼저 읽습니다.
    1. `Documents/Coding_Rules.md` — 코딩 규칙 및 컨벤션 (항상 최우선 적용)
    2. 작업과 직접 관련된 `Documents/` 내 기타 문서 (설계서, 기획서 등)
*   분석한 내용을 기반으로 작업 방향성, 데이터 흐름, 예외 처리 등을 정의한 세부 구현 계획을 먼저 수립합니다.

### 2단계: 코드 효율성 및 재사용성 검토
*   새로운 컴포넌트나 함수를 무작정 설계하기 전에, 프로젝트 공통 유틸리티(`utils.ts`), 공유 훅(`useEditorFormat.ts`), 혹은 기구현된 하위 컴포넌트에서 재사용할 수 있는 로직이 있는지 우선 확인합니다.
*   기존 모듈을 최대한 활용 및 확장하여 코드 중복을 억제하고 가장 효율적인 코딩 설계를 지향합니다.

### 3단계: 작업 영역 및 영향 범위 식별
*   수정이 불가피한 영역(파일 경로, 컴포넌트 내부 코드 블록)을 정확히 정밀 검사하여 식별합니다.
*   신규 구현이 기존의 타입 시스템이나 의존 관계에 영향을 미치는지 사전에 분석하여 부작용을 예방합니다.

### 4단계: 타깃 영역 정밀 수정 및 사용자 코드 보존
*   사용자가 명시적으로 지시한 요구사항 및 이와 연계된 직접적 연관 부분만 정밀 수정(Surgical Edit)합니다.
*   **범위 외 코드 수정 금지**: 지시한 작업과 직접적인 인과관계가 없는 코드는 설령 개선 여지가 보이더라도 **절대 임의로 수정하지 않습니다.** 수정이 필요하다고 판단될 경우, 수정 이유와 범위를 사용자에게 먼저 설명하고 허가를 받은 후에만 진행합니다.
*   **사용자 수정 코드 허가 필수**: 사용자가 직접 수정하거나 작성한 코드 영역은 **에이전트가 임의로 변경할 수 없습니다.** 해당 부분의 수정이 필요할 경우:
    1. 해당 코드가 어디에 있는지와 수정이 필요한 이유를 사용자에게 명확히 설명
    2. 사용자의 명시적 허가(승인)를 받은 후에만 수정 진행
    3. 단, 해당 코드가 빌드를 파괴하는 치명적 컴파일 에러를 유발할 경우에만 예외로 최소 처리 후 즉시 보고
*   **작업 보고 명세**: 작업 완료 후 보고할 때, 사용자가 작성하여 남겨둔 코드 부분에 대한 처리 내용이나 현황을 상세히 요약하여 함께 고지합니다.

## 2. 효율적이고 안정적인 코딩을 위한 세부 추가 규칙

### [안정성 & 정밀성 규정]
*   **컴파일 무결성 보장**: 코드 수정 후에는 반드시 타입스크립트 빌드(`npx tsc -b`)와 린트(`npm run lint`)를 수행하여 오류가 없는지 검증합니다.
*   **점진적 수정 및 병합**: 여러 파일에 걸쳐 수정할 때 대규모 영역을 뭉텅이로 치환하지 않고, 줄 단위로 정교하게 비교/대체하여 기존에 정상 작동하던 기능이 지워지는 일을 완벽히 방지합니다.
*   **에러 발생 시 피드백**: 만약 사용자가 임시로 작성해 둔 코드가 현재 작업 구현체와 부딪혀 치명적인 컴파일 에러를 유발하는 경우, 임의로 지우지 않고 에러 발생 위치와 해결안을 리포트하여 조율하거나 최소한의 에러 예방 처리만 더합니다.
*   **비파괴적 인터페이스 유지**: 기존 컴포넌트들의 인터페이스(Props 규격 등)를 수정해야 할 경우, 기존 호출부에서 오류가 나지 않도록 기본값 제공 또는 선택적 프로퍼티(`?`) 처리를 우선 고려합니다.
*   **오픈 소스 라이브러리 활용 규정**: 새로운 기능을 구현하거나 개편할 경우 무조건 제로 베이스에서 처음부터 직접 개발하지 않고, 인터넷 조사를 통해 상업적 이용이 가능한 무료 오픈 소스 라이브러리(MIT, Apache 2.0 등)가 존재한다면 적극적으로 활용합니다. 단, 라이브러리에 저작권자 표기(Attribution)나 특수 조건이 있을 경우 구현 전에 반드시 사용자에게 공지하고 승인을 얻어야 합니다.
